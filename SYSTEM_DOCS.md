## 1. System Overview

A full-featured Hebrew-language CRM for a chocolate/petit-four patisserie ("עדי תכשיט שוקולד"). The system manages:

- **Customers** — profiles, type tiers, discount rules, communication history
- **Orders** — creation, items, petit-four box selections, pricing, discounts
- **Deliveries** — courier assignment, WhatsApp/email auto-notification, token-based confirmation page
- **Products** — regular products and petit-four boxes; business-only restriction
- **Inventory** — raw materials with threshold-based status alerts
- **Recipes** — link products to raw-material consumption; auto-deduct on production
- **Production** — log batch production; auto-deduct raw materials via recipes
- **Invoices & Payments** — linked to orders
- **Communications** — log of emails and WhatsApp messages per customer
- **Import** — CSV/JSON bulk import for products, customers, orders
- **Settings** — business name, logo, contact info

The system is deployed on **Vercel** and uses **Supabase** (PostgreSQL + Storage) as backend. All API routes are Next.js App Router Route Handlers.

## 2. Architecture

### Frontend
- **Framework**: Next.js 14 (App Router, `use client` where needed)
- **Styling**: Tailwind CSS + inline styles; warm brown/gold luxury palette
- **RTL**: `direction: rtl` set globally in `globals.css` and on `<html>`
- **State**: Component-level `useState` / `useEffect` — no global state manager
- **Notifications**: `react-hot-toast`
- **Export**: Custom `exportToCsv` utility

### Backend
- **API**: Next.js Route Handlers under `src/app/api/`
- **Auth**: None implemented — all routes are open (admin-only assumed context)
- **DB client**: Supabase JS client via `createAdminClient()` (service-role key, bypasses RLS)
- **Email**: SendGrid (`@sendgrid/mail`) for courier/customer emails; Resend (`resend`) for order summaries
- **File storage**: Supabase Storage

### Database
- **PostgreSQL** via Supabase
- All table and column names are in **Hebrew** (UTF-8), except a small number of English columns added in later migrations (`courier_id`, `delivery_token`, `delivered_at`, `whatsapp_sent_at`, `email_sent_at`, `created_at`, `updated_at` on the couriers table)
- Auto-updated `תאריך_עדכון` via Postgres triggers on most tables
- Inventory status (`סטטוס_מלאי`) is auto-computed by a Postgres trigger on every insert/update

## 5. API Endpoints

All routes are under `/api/`. All return JSON. All use `createAdminClient()` (service-role Supabase).

### 5.1 Dashboard

#### `GET /api/dashboard`

Returns real-time business metrics. Uses `Asia/Jerusalem` timezone.

```json
{
  "ordersToday": 3,
  "ordersTomorrow": 1,
  "urgentOrders": 2,
  "unpaidOrders": 5,
  "inPreparation": 4,
  "readyForDelivery": 1,
  "shipped": 0,
  "lowInventory": 2,
  "deliveriesToday": 3,
  "unpaidAmount": 1450.00
}
```

### 5.2 Customers

#### `GET /api/customers`

Query params: `search` (name/phone/email), `type` (`פרטי`|`חוזר`|`עסקי`), `limit` (default 100), `offset`.

```json
{
  "data": [
    {
      "id": "uuid",
      "שם_פרטי": "Sarah",
      "שם_משפחה": "Cohen",
      "טלפון": "050-1234567",
      "אימייל": "sarah@example.com",
      "סוג_לקוח": "חוזר",
      "אחוז_הנחה": 10,
      "תאריך_יצירה": "2024-01-15T10:00:00Z"
    }
  ]
}
```

#### `POST /api/customers`

Required: `שם_פרטי`. Returns 201 with created record.

#### `GET /api/customers/[id]`

Returns customer + joined: orders, invoices, uploaded_files, תיעוד_תקשורת.

#### `PATCH /api/customers/[id]`

Partial update. All fields optional.

#### `DELETE /api/customers/[id]`

Returns 400 if customer has orders. Prevents orphan data.

#### `GET /api/customers/[id]/communication`

Returns communication logs newest-first.

#### `POST /api/customers/[id]/communication`

Logs a manual communication entry.

```json
{
  "סוג": "וואטסאפ",
  "תוכן": "Confirmed delivery for tomorrow",
  "הזמנה_id": "uuid-optional"
}
```

#### `POST /api/customers/[id]/send-email`

Sends email via SendGrid and logs the result to `תיעוד_תקשורת`.

```json
{
  "subject": "Your order is ready",
  "content": "Dear Sarah, your chocolate box is ready for pickup."
}
```

### 5.11 Deliveries

#### `GET /api/deliveries`

**Two modes:**

**Archive mode** (`?mode=archive`): Returns all deliveries with `סטטוס_משלוח = 'נמסר'`, with joined order and courier data.

**Active mode** (default): 
1. Fetches all qualifying orders: `סוג_אספקה = 'משלוח'` OR `כתובת_מקבל_ההזמנה IS NOT NULL` OR `דמי_משלוח > 0`
2. Excludes archived/cancelled orders
3. Optional `?date=YYYY-MM-DD` filter
4. Optional `?status=` filter
5. Joins with delivery records. Orders without a record get a **synthetic row** with `_noRecord: true` and `id: "no-record-{orderId}"`

```json
{
  "data": [
    {
      "id": "uuid-real",
      "הזמנה_id": "uuid",
      "סטטוס_משלוח": "ממתין",
      "courier_id": "uuid",
      "delivery_token": null,
      "_noRecord": false,
      "הזמנות": { "מספר_הזמנה": "ORD-0042", "שם_מקבל": "Sarah" },
      "שליחים": { "id": "uuid", "שם_שליח": "David", "טלפון_שליח": "052-..." }
    },
    {
      "id": "no-record-order-uuid",
      "_noRecord": true,
      "סטטוס_משלוח": "ממתין",
      "הזמנה_id": "order-uuid"
    }
  ]
}
```

#### `POST /api/deliveries`

Creates a delivery record manually. Required: `הזמנה_id`.

#### `PATCH /api/deliveries/[id]`

**Smart PATCH** — handles three scenarios:

1. **Synthetic ID** (`id` starts with `no-record-`):
   - Checks if a delivery already exists for the order (race guard)
   - If not, fetches order data and auto-creates a delivery record
   - Continues with the real delivery ID

2. **Status change to `נאסף`**:
   - Fetches delivery + courier via wildcard join `שליחים!courier_id(*)`
   - Generates `delivery_token` if missing (32-byte hex via `crypto.randomBytes`)
   - Builds delivery confirmation URL: `{origin}/delivery-update/{token}`
   - Resolves recipient name from order
   - Sends notification based on courier contact:
     - **Phone** → builds `wa.me` WhatsApp URL, returns it in response (frontend opens it), sets `whatsapp_sent_at`
     - **Email only** → sends via SendGrid, sets `email_sent_at`
     - **Neither** → returns `send_error`
   - Checks `whatsapp_sent_at` / `email_sent_at` to prevent duplicate sends

3. **Any other update** (courier assignment, notes, etc.):
   - Plain `UPDATE` on the delivery record

Response for `נאסף` transition:
```json
{
  "data": { "id": "uuid", "סטטוס_משלוח": "נאסף", "delivery_token": "abc123...", ... },
  "token": "abc123...",
  "delivery_link": "https://yourapp.com/delivery-update/abc123...",
  "whatsapp_url": "https://wa.me/972529876543?text=...",
  "auto_sent": "whatsapp",
  "send_error": null,
  "was_created": false
}
```

#### `POST /api/deliveries/[id]/token`

Generates or retrieves a delivery token. Returns `{ token, link }`.

#### `POST /api/deliveries/[id]/send-courier-email`

Manual trigger: sends a SendGrid email to the courier's `אימייל_שליח` with a "Mark as Delivered" button linking to `/delivery-update/{token}`. Sets `email_sent_at`.

## 6. Core Flows

### 6.1 Creating a New Order (Full Flow)

1. User navigates to `/orders/new`
2. Selects existing customer OR fills in new customer details
3. Fills order metadata (date, delivery type, address, urgency flag, blessing text)
4. Adds items:
   - Regular product: select from dropdown, enter quantity
   - Petit-four box: select box size, then select flavors and quantities per flavor
5. System calculates totals with discount (% or fixed amount)
6. If any item has `לקוחות_עסקיים_בלבד = true` and customer is not `'עסקי'` → item is hidden from the dropdown (frontend) and rejected with 403 (backend)
7. `POST /api/orders/create-full` is called atomically:
   - Creates/finds customer
   - Creates order with calculated totals
   - Creates order items + petit-four selections
   - Creates delivery record (if delivery type = `משלוח`)
   - Creates initial payment record
8. On success → redirects to `/orders/[id]`

### 6.2 Delivery Notification Flow

1. User goes to `/deliveries`
2. Assigns a courier to the delivery using the courier dropdown (calls `PATCH /api/deliveries/[id]` with `courier_id`)
3. Changes delivery status to `'נאסף'`
4. Backend auto-creates delivery record if one doesn't exist (synthetic ID handling)
5. Backend generates `delivery_token` if missing
6. Backend selects notification channel:
   - Courier has phone → builds WhatsApp `wa.me` URL → returns `whatsapp_url` to frontend → frontend calls `window.open(whatsapp_url, '_blank')` to open WhatsApp with pre-filled message
   - Courier has email only → backend sends email via SendGrid with "Mark as Delivered" button linking to `/delivery-update/{token}`
   - Courier has both → frontend shows channel-choice modal, user picks one
7. `whatsapp_sent_at` or `email_sent_at` is stored to prevent duplicate sends
8. Courier taps the link → public page at `/delivery-update/{token}` (not yet built)
9. Courier confirms delivery → `POST /api/delivery-update/[token]` → status becomes `'נמסר'`, `delivered_at` is set
10. Delivery disappears from active list, appears in Archive tab

### 6.3 Inventory Deduction

**On order status change (חדשה → בהכנה)**:
- `PATCH /api/orders/[id]` detects first transition
- Decrements `כמות_במלאי` on each product in the order

**On production log creation**:
- `POST /api/production` finds the recipe linked to the product
- Scales ingredient quantities: `required = ingredient.כמות_נדרשת × (produced / recipe.כמות_תוצר)`
- Deducts from `מלאי_חומרי_גלם.כמות_במלאי`
- Postgres trigger automatically recalculates `סטטוס_מלאי`

### 6.4 Customer Discount Application

- `POST /api/orders/create-full` reads `לקוחות.אחוז_הנחה`
- Automatically pre-fills `סוג_הנחה = 'אחוז'` and `ערך_הנחה = customer.אחוז_הנחה` if customer has a discount
- User can override type (%, fixed amount, or none) and value before submitting
- Final totals: `סכום_לפני_הנחה` = sum of items + delivery fee; `סכום_הנחה` = calculated deduction; `סך_הכל_לתשלום` = final

## 7. Business Logic

### 7.1 Order Statuses

| Status (Hebrew) | Meaning | Notes |
|---|---|---|
| `חדשה` | New | Initial state |
| `בהכנה` | In preparation | Triggers inventory deduction on first transition |
| `מוכנה למשלוח` | Ready for delivery | |
| `נשלחה` | Shipped/sent | |
| `הושלמה בהצלחה` | Completed | Sets `ארכיון = true`; excluded from active lists |
| `בוטלה` | Cancelled | Excluded from active lists |

### 7.2 Delivery Statuses

| Status | Meaning |
|---|---|
| `ממתין` | Awaiting pickup by courier |
| `נאסף` | Picked up by courier, in transit |
| `נמסר` | Delivered to recipient |

Transition to `נאסף` triggers automated courier notification. Transition to `נמסר` sets `delivered_at` and moves delivery to archive.

### 7.3 Customer Types

| Type | Meaning | Notes |
|---|---|---|
| `פרטי` | Private individual | Default |
| `חוזר` | Returning customer | Gets display badge |
| `עסקי` | Business client | Can access business-only products |

### 7.4 Business-Only Products

- Products with `לקוחות_עסקיים_בלבד = true` are:
  - Hidden from product dropdowns for non-business customers (frontend filter)
  - Rejected at server with 403 if order is placed by a non-`עסקי` customer

### 7.5 Discount Types

| Type | Meaning | Calculation |
|---|---|---|
| `ללא` | No discount | `סכום_הנחה = 0` |
| `אחוז` | Percentage | `סכום_הנחה = סכום_לפני_הנחה × (ערך_הנחה / 100)` |
| `סכום` | Fixed amount | `סכום_הנחה = ערך_הנחה` |

### 7.6 Inventory Status (Auto-Computed by Trigger)

| Status | Condition |
|---|---|
| `אזל מהמלאי` | `כמות_במלאי <= 0` |
| `קריטי` | `כמות_במלאי <= סף_מלאי_קריטי` |
| `מלאי נמוך` | `כמות_במלאי <= סף_מלאי_נמוך` |
| `תקין` | All thresholds OK |

### 7.7 Delivery Eligibility

An order qualifies as a delivery (appears in the deliveries page) if ANY of:
- `סוג_אספקה = 'משלוח'`
- `כתובת_מקבל_ההזמנה IS NOT NULL`
- `דמי_משלוח > 0`

Orders with `סטטוס_הזמנה IN ('הושלמה בהצלחה', 'בוטלה')` are excluded from the active deliveries list.

### 7.8 Courier Notification Priority

When status changes to `נאסף`:
1. If courier has `טלפון_שליח` → WhatsApp (regardless of email)
2. If courier has only `אימייל_שליח` → Email via SendGrid
3. If both exist → Frontend choice modal (user picks channel)
4. If neither → `send_error` returned, delivery status still saved

Duplicate-send prevention: `whatsapp_sent_at` and `email_sent_at` are checked before sending. If already set → skip.

## 9. Known Issues

### 9.1 Migrations Must Be Run Manually

The 10 migration files in `supabase/migrations/` are **not auto-applied**. They must be run manually in the Supabase SQL Editor. Particularly critical:
- `008_add_courier_email.sql` — adds `אימייל_שליח` on couriers and `email_sent_at` on deliveries. If not run, the courier email feature silently fails.
- `006_add_discount_type.sql` — adds `סוג_הנחה` and `ערך_הנחה` columns. If not run, discount-type orders fail to save.

The code is resilient for some of these (uses wildcard joins, conditional column inclusion) but not all.

### 9.2 Public Delivery Confirmation Page Not Built

The API routes `GET /api/delivery-update/[token]` and `POST /api/delivery-update/[token]` are implemented. The frontend page at `/delivery-update/[token]/page.tsx` **does not exist**. Couriers receive a link that leads to a 404.

### 9.3 Email Sending Shows Success Without Delivery

If `SENDGRID_API_KEY` or `FROM_EMAIL` are missing in Vercel, the system shows a success toast for email sending but the email is never sent. Added in latest commits: detailed server-side logging (`console.log`) for ENV var presence and SendGrid response status. Check Vercel function logs to diagnose. Also verify:
- `FROM_EMAIL` is a verified Sender Identity in SendGrid
- SendGrid Sandbox Mode is **off** (Settings → Mail Settings)

### 9.4 No Authentication

The entire application has no auth layer. Any user who knows the URL can access all data. Suitable for single-user/single-device deployments only. Adding Supabase Auth or NextAuth.js is the first priority for any multi-user scenario.

### 9.5 Duplicate Migration Numbers

Files `006_add_discount_type.sql` and `006_business_only_products.sql` both start with `006_`. Same for `007_add_couriers.sql` and `007_simplify_customer_types.sql`. All four must be run. There is no migration runner — just run them all in Supabase SQL Editor.

### 9.6 Delivery Update Page (Public) — No Error UI

When a courier visits the delivery link, if the token is invalid or expired, there is no friendly error page since the page doesn't exist yet.

### 9.7 Inventory Deduction on Order Status Change Is One-Way

When an order transitions from `חדשה` → `בהכנה`, product stock is decremented. If the order is later cancelled, stock is **not** restored automatically. This must be done manually.
